## 6. 新增规划：主界面 / 周界面右键上下文菜单

### 6.1 功能目标

主界面右键菜单：

- 在主界面日程卡片区域、空状态区域或日程列表容器区域右键弹出上下文菜单。
- 菜单项与顶部按键组概念一致：
  - 换肤
  - 视图
  - 添加
  - 排序
  - 筛选
- 本阶段先实装已存在功能：
  - `视图`
  - `添加`
- 未实装功能保留占位：
  - 换肤
  - 排序
  - 筛选

周界面右键菜单：

- 在周界面某日期对应的白色空白区域右键弹出上下文菜单。
- 菜单项与主界面保持一致。
- `添加` 应使用右键所在日期作为添加日期。
- 过去日期不能添加日程。
- `视图` 菜单与主界面一致。

视图子菜单：

- 点击主菜单中的 `视图` 时，在右侧弹出紧邻的小子菜单。
- 子菜单项：
  - 日视图
  - 周视图
  - 月视图
  - 四象限视图
  - 待办
- 已实装路由：
  - 日视图
  - 周视图
  - 月视图
  - 待办
- 四象限视图暂不实现，保持禁用或占位提示，不伪实现。

视觉约束：

- 布局参考现有更多菜单：左侧图标，右侧功能文字。
- 图标优先复用 `assets/icons/` 中已有图标：
  - `Skin.svg` 或 `theme.svg`
  - `view.svg`
  - `add.svg`
  - `sort.svg`
  - `filter.svg`
  - `Calendar.svg`
  - `week_top_color.svg` 或现有周视图图标替代
  - `todo.svg`
- 不新增位图资源，除非后续小工单明确要求。

### 6.2 架构方向

右键菜单不应直接大段塞进 `MainWindow` 或 `WeekWindow`。

建议方向：

- 先建立只读基线，明确右键目标区域、已有顶部按钮动作、已有菜单组件能力。
- 再根据 `CM-0` 结论新增或最小复用/扩展一个小型公共菜单组件。例如：
  - `src/ui/common/action_context_menu.py`
- 若现有 `SharedMoreMenu` / `ScheduleContextMenu` 不适合作为通用右键菜单，再新增独立组件。
- 公共组件只负责菜单 UI 和 action 信号，不直接写数据库，不直接访问 `MainWindow` / `WeekWindow` 业务状态。
- 主界面和周界面负责把菜单信号接到已有动作：
  - 主界面添加：复用 `MainWindow.switch_to_add_page()`
  - 主界面视图：复用 `MainWindow.switch_view(...)`
  - 周界面添加：先设置 `WeekWindow.current_selected_date`，再复用 `WeekWindow.switch_to_add_page()`
  - 周界面视图：复用已有 `WeekWindow.view_selected.emit(...)` 或现有视图切换路径
- 不复制 `ViewRouter` 中的视图识别规则；需要识别 view_name 时复用已有路由入口。

### 6.3 风险点

低风险：

- 新建只展示菜单的公共组件。
- 主界面右键菜单中 `添加` 调用已有 `switch_to_add_page()`。
- 主界面右键菜单中 `视图` 子菜单调用已有 `switch_view(...)`。

中风险：

- 周界面白色空白区域右键定位到具体日期。
- 周界面右键添加需要正确设置 `current_selected_date`。
- 右键菜单和现有拖拽、点击选中、双击跳转共存。

高风险暂缓：

- 换肤真实 UI。
- 排序完整功能。
- 筛选完整功能。
- 四象限视图。
- 统一替换所有菜单组件。
- 大范围 QSS 菜单样式迁移。

## 8. 执行策略

- 每个小工单只允许小范围修改。
- 涉及 UI 行为变化时，必须先记录旧行为。
- 涉及跨视图跳转时，优先复用已有方法，不复制逻辑。
- 涉及右键菜单时，先确认事件不会破坏左键、双击、拖拽、卡片右键菜单。
- 涉及周界面日期时，必须确认 QDate / Python date 转换边界。
- 涉及数据库写入时，必须明确是否会改 `schedule.db`，并在验收后确认无非预期 tracked diff。
- 每个执行任务完成后等待主窗口复核，不直接提交 Git。
