# Workflow Guide

用途：定义当前项目后续功能补充阶段的三窗口协作流程。

当前协作角色：

- 主窗口：调度、审核、决策收口、风险判断、最终提示词输出。
- 决策窗口：提出方案、拆分工单、补充风险和验收建议。
- 执行窗口：按最终提示词执行具体任务、记录日志、等待复核。

---

## 1. 基本原则

- 主窗口负责最终把关，不能把决策窗口输出原样交给执行窗口。
- 决策窗口只提出方案，不直接改源码。
- 执行窗口只执行主窗口确认后的最终提示词。
- 每个小工单必须有明确允许/禁止范围。
- 涉及 UI 行为、数据库、刷新链路、picker、弹窗回流时，优先只读审查，再执行修改。
- 当前已结束架构改写主线，后续进入功能补充与局部完善阶段。

---

## 2. 标准工作流

### 2.1 规划

1. 用户提出阶段方向或功能目标。
2. 主窗口读取当前文档和代码状态。
3. 如目标复杂，主窗口要求决策窗口生成方案。
4. 主窗口审核方案，必要时改写。
5. 主窗口把最终规划写入 `Work_Formulation.md`。

### 2.2 拆分

1. 主窗口把规划拆成小工单。
2. 当前待执行工单写入 `Work_Instruction.md`。
3. 面向执行窗口的最终提示词写入 `Work_Task_Prompts.md`。
4. 最终提示词必须在主窗口直接发给用户。

### 2.3 执行

1. 执行窗口按 `Work_Task_Prompts.md` 执行。
2. 执行窗口更新 `Work_Log.md`。
3. 执行窗口不得提交 Git，除非用户明确要求。
4. 用户把执行结果带回主窗口。

### 2.4 复核

1. 主窗口读取 diff 和日志。
2. 主窗口复跑关键验收命令。
3. 如发现缺口，主窗口修正日志或生成补充提示词。
4. 通过后，主窗口给出提交命令。

### 2.5 归档

1. 阶段完成后，主窗口把阶段指令和日志归档到 `History_Instruction.md` / `History_Log.md`。
2. `Work_Instruction.md`、`Work_Log.md`、`Work_Task_Prompts.md` 重置为下一阶段待规划状态。

## 3. 当前阶段重点

当前项目进入功能补充阶段。近期优先级：

1. 完善月界面功能。
2. 审核并实现周界面日期双击跳转日视图等低风险交互增强。
3. 完成月界面回归验收后，再制定后续功能补充规划。

暂不优先实现：

- 四象限视图。
- 云同步完整闭环。
- 大范围 UI 重构。
- 大范围样式迁移。
