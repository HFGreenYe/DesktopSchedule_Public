用途：沉淀项目最终架构目标、当前阶段功能开发约束和已累积技术债。后续新功能规划应优先读取本文，再读取当前 `Work_Formulation.md` / `Work_Instruction.md`。

## 1. 总体方向

项目采用兼容式渐进演进，不重写产品，不一次性替换旧 UI。

核心原则：

- 保留现有运行行为优先于目录形式整齐。
- 保留 `db_manager` 作为旧 UI 过渡门面。
- 新增功能优先旁路接入，不破坏旧调用链。
- 新公共 UI 组件优先放入 `src/ui/common/`、`src/ui/popups/` 或 `src/ui/utils/`。
- 新功能不得把大量业务判断继续堆入窗口类；能抽纯逻辑时应进入 service 或 helper。
- 占位功能必须保持占位，不伪实现。
- 每个阶段继续采用小工单推进，先审查、再单点试点、最后整体验收。

## 2. 最终目标架构

长期目标不是推翻当前结构，而是逐步形成以下分层：

```text
src/
  app/ or controllers/
    bootstrap.py              # QApplication、窗口创建、全局装配
    main_controller.py         # 主窗口级路由与跨视图协调
    view_router.py             # day/week/month/todo/add/picker 切换
    refresh_coordinator.py     # 跨视图刷新协调

  domain/ or models/
    models.py                  # 轻量领域对象、枚举、状态常量
    schedule_rules.py          # 重复日程、日期计算、过期判断
    sorting.py                 # 日程/待办排序规则
    reminders.py               # 提醒触发判断

  data/
    models.py                  # Peewee Schedule/Category
    database.py                # db 连接、初始化、迁移、db_manager 兼容门面

  repositories/
    schedule_repository.py     # 日程 CRUD
    category_repository.py     # 清单 CRUD

  services/
    schedule_query_service.py          # 日程/待办过滤
    schedule_sort_service.py           # 排序规则
    schedule_service.py                # 新建、编辑、重复日程写入
    category_policy_service.py         # 清单删除策略
    reminder_service.py                # 提醒扫描与触发判断
    schedule_axis_service.py           # 坐标看板时间投影、范围与对数映射纯逻辑
    weather_service.py                 # 天气服务

  theme/
    theme_manager.py           # Skin Preset / QSS 加载与刷新
    default.qss                # 默认皮肤
    *.qss                      # 后续皮肤

  ui/
    common/                    # 可复用 UI 小组件
    popups/                    # 独立弹窗/浮窗
    utils/                     # UI 工具 helper
    views/                     # 后续视图拆分目标
    dialogs/                   # 后续对话框拆分目标
    *.py                       # 现有兼容 UI 文件
```

当前项目已经采用实际目录：

- `src/controllers/`
- `src/repositories/`
- `src/services/`
- `src/theme/`
- `src/ui/common/`
- `src/ui/popups/`
- `src/ui/utils/`

后续不必为了和旧规划的 `app/`、`domain/` 名称完全一致而重排目录；实际原则是职责边界一致。

## 4. 后续功能开发约束

新增功能必须遵守：

- 不新增数据库字段，除非阶段规划明确说明并单独验收迁移。
- UI 层可以短期继续通过 `db_manager` 兼容门面读取数据，但不应直接新增 Repository 依赖。
- 新增复杂业务规则优先进入 `services/`，不要写进窗口类。
- 新弹窗优先进入 `src/ui/popups/`。
- 可复用小组件优先进入 `src/ui/common/`。
- 纯 UI helper 优先进入 `src/ui/utils/`。
- 新右键菜单优先复用 `ActionContextMenu`。
- 新视图样式应优先考虑 `default.qss / skin preset / role-state-variant`，旧 UI 内联样式暂不强行清理。
- 所有功能阶段都必须先写规划，再写分步指令，再生成提示词。

## 6. 当前技术债总表

### 6.1 Data / Repository / Service

- `db_manager` 仍承担大量旧 UI 兼容入口。
- 部分 UI 仍直接调用 `db_manager` 做查询或写入。
- 月界面当前在 `MonthWindow` 内部构建 marker cache 和 hover cache，短期可接受，长期应考虑提到 service/read-model helper。
- 重复日程写入和编辑逻辑仍属于高风险区域，后续修改必须单独小工单验证。

建议：

- 保持 `db_manager` 门面不动。
- 后续新增复杂查询优先进入 `ScheduleQueryService` 或专门 read-model helper。
- 不在功能工单中顺手重构重复规则写入。

### 6.2 Controller / Router / Refresh

建议：

### 6.6 Weather

已处理的基础兜底：

- `weather_service` 已对 API 配置缺失、HTTP 错误、空响应、非 JSON、定位失败、天气 API code 异常和关键字段缺失做显式校验。
- 失败时不再向 UI 发空 dict，而是发结构完整的 fallback 数据：
  - `temp = "--"`
  - `icon = "999"`
  - `text = "天气暂不可用"`
  - `city = "未知位置"`
  - `available = False`
  - `error = <失败原因>`
- 成功数据增加 `available = True`。
- `HeaderBar`、`WeekWindow`、`MonthWindow` 初始化等待阶段使用 `assets/icons/hourglass.svg` 做上下翻转过渡，最长 3000ms。
- 沙漏在独立 loading label 内按 78% 尺寸居中绘制，并围绕画布几何中心水平线做上下翻转；weather label 独立显示真实天气 / `999-fill.svg`，避免过渡动画影响正常天气图标位置。
- 若 3000ms 内未收到天气结果，UI 自动切换到 `assets/weather/999-fill.svg`。
- 若 loading 期间先收到 fallback `999-fill.svg`，仍等到 3000ms 后切换；真实天气数据返回时立即停止 loading 并显示真实天气 SVG。
- 若后续收到真实天气数据，UI 停止加载态并显示 API 对应天气 SVG。
- 天气图标加载失败时使用 `assets/weather/999-fill.svg`，不再使用天气 emoji 或普通问号作为中间态。
- 如果 `999-fill.svg` 也异常缺失，则清空天气图标，不再显示普通问号。

仍未实现：

- 不自动重试。
- 不缓存上一次成功天气。
- 不切换备用天气源。
- 不提供用户可见的详细错误面板。

建议：

- 若后续继续增强天气模块，应单独开小工单处理重试、缓存上次成功天气或备用天气源。
- 天气增强不应混入月界面 UI 适配、坐标看板或导出功能阶段。

   - 复查 DDL 文本策略、日程块省略号、选中列高亮和悬停预览在高 DPI / 不同窗口宽度下是否仍可接受。

3. 搜索 / 筛选功能。

   补齐日程和待办的搜索、筛选入口，优先覆盖标题、日期范围、清单、完成状态、紧急性等基础条件。

4. 实现导出功能。

   导出功能建议分阶段实现，先做结构稳定、低风险的 Markdown，再扩展 PDF 和 PNG。

   目标能力：

   - 支持导出内容类型：
     - 仅日程。
     - 仅待办。
     - 日程 + 待办。
   - 支持日程导出范围：
     - 某日。
     - 某月。
     - 某年。
     - 全部。
   - 支持导出格式：
     - Markdown (`.md`)。
     - PDF (`.pdf`)。
     - PNG (`.png`)。

   推荐实现顺序：

   - `Export-0`：导出需求与数据字段只读审查。

     确认日程、待办、分类、提醒、重复规则、完成状态、紧急性字段的真实存储和显示语义。

   - `Export-1`：建立导出数据模型 / 中间结构。

     建立统一 `ExportPayload` 或等价结构，避免 Markdown / PDF / PNG 各自重复查库和拼数据。

   - `Export-2`：Markdown 导出。

     先实现 `.md` 导出，按日期或清单分组，包含时间、标题、状态、紧急性、重复、提醒、所属清单等基础信息。

   - `Export-3`：导出入口 UI。

     增加导出入口和导出选项，支持选择内容类型、时间范围和导出路径。

   - `Export-4`：PDF 导出。

     基于同一份导出数据模型生成 PDF。优先保证文字内容、分页和中文字体稳定，不追求复杂排版。

   - `Export-5`：PNG 导出。

     PNG 先作为摘要图或海报图能力，不建议一开始支持“全部数据”长图。优先支持某日 / 某月摘要，可后续再支持背景图设置。

   边界约束：

   - 不新增数据库字段。
   - 不修改日程 / 待办写入逻辑。
   - 不把导出逻辑写进大型窗口类。
   - 数据查询和格式组装应放入 service / helper。
   - UI 只负责选择导出范围、格式和保存路径。
   - PNG 背景图属于后续增强，不阻塞 Markdown / PDF。
   - 导出功能可作为后续本地备份 / 云同步的数据结构参考，但本阶段不实现备份恢复和云同步。

5. 自动化测试与 CI 基线。

   在搜索和导出阶段完成后集中建立测试基线，至少覆盖重复规则、查询 / 导出中间结构、坐标映射、Repository 临时数据库、跨视图路由刷新和关键 PyQt smoke。该阶段不追求旧 UI 全覆盖，但必须为后续自定义重复规则提供可重复执行的回归保护。

6. 自定义重复规则。

   在现有“无 / 日 / 周 / 月”等固定重复规则之外，后续增加“自定义”规则。点击“自定义”后打开日历弹窗，允许用户单击选择任意多个日期，也允许按住鼠标左键滑动选取一段连续日期。实现前必须先审查当前 `repeat_rule`、重复日程写入和编辑逻辑，确认是否需要新增存储结构或采用兼容字段，不得在 UI 阶段直接改数据库结构。

7. 实现皮肤功能。

   基于现有 ThemeManager / Skin Preset，接入真实换肤入口，逐步处理内联样式覆盖问题。

   皮肤配置采用“内置默认规则 + JSON 语义令牌局部覆盖 + QSS / 自绘组件统一消费”的分层方案：

   - `ThemeManager` 继续负责 Skin Preset、QSS 加载、主题切换和控件刷新；JSON 不替代 QSS，而是作为 QSS 和自绘控件共同使用的主题数据层。
   - 内置默认主题必须提供完整语义令牌，例如窗口渐变上沿/下沿、主次文字、输入框背景、边框、菜单、弹窗和按钮状态颜色。
   - 普通渐变或纯色皮肤可以只覆盖基础背景令牌；未显式设置的派生颜色按默认算法生成。例如详情输入框默认读取渐变上沿颜色，通过亮度降低和透明度规则生成协调背景；纯色主题只需令渐变上下沿相同。
   - 复杂撞色皮肤允许在 JSON 中显式覆盖 `detail_editor.background` 等具体语义令牌；JSON 只需填写需要修改的字段，缺失字段继续继承默认主题。
   - 合并顺序固定为：内置默认值 -> base preset 基础令牌 -> JSON 基础令牌覆盖 -> 派生语义颜色计算 -> JSON 显式语义覆盖。这样修改主色时派生组件自动协调，特殊设计仍可覆盖计算结果。
   - JSON 应包含 `schema_version` 和可选 `base`，颜色、尺寸、字体、图标等字段必须走白名单和格式校验；未知或非法字段忽略并回退默认值，不允许 JSON 注入任意 QSS。
   - 主题运行时应形成统一的 Theme Tokens 对象，供 QSS 模板、`StyleManager` 和 `paintEvent` 自绘组件读取，避免继续从 `AppConfig`、内联 QSS 和局部硬编码分别取色。
   - 当前 `StyleManager.derive_surface_rgba(...)` 与详情框动态背景可作为首个派生语义色样本；正式换肤阶段再迁移其他内联样式，不在普通功能轮批量重写。

   皮肤功能之后的暗色模式专项规划：

   - 暗色模式不再按“某个窗口临时改黑 / 改白”推进。当前周界面暗色试验已证明问题主要来自内联 QSS、自绘 `paintEvent`、SVG / PNG 图标 mask、详情弹窗和局部容器背景各自取色，继续点修容易出现漏项、遮罩误加、文字颜色不一致和选中高光断层。
   - 暗色模式应建立在皮肤功能的 Theme Tokens 之上，作为一个内置 skin preset 或 `base=dark` 的主题变体，而不是独立散落的 `_dark_mode` 特判矩阵。短期可保留 `_dark_mode` 作为兼容开关，但实现层应逐步改为读取统一 token。
   - 必须先定义暗色语义令牌：窗口渐变上沿 / 下沿、主背景、内容面板、选中列高光、输入框背景、详情编辑框背景、主文字、次文字、禁用文字、边框、分隔线、菜单、按钮 hover / pressed、日程块文字、完成 / 过期状态、当前时间线和图标前景色。
   - 自绘组件必须纳入统一消费范围：日 / 周 / 月窗口、日课表、周课表、坐标看板、月格 marker、详情弹窗、更多菜单、picker、颜色弹窗、hover 预览和挂起条。凡是 `paintEvent` 或动态 `QPainter` 绘制的颜色，都不应继续硬编码。
   - 图标规则必须单独处理：SVG 优先走 `get_colored_icon(...)` 或统一 icon tint 工具；PNG 图标需要明确 mask / tint 路径；天气 SVG 需保留当前图标路径并支持主题切换后立即重染，不依赖天气服务刷新。
   - 背景和透明度必须分层：主窗口渐变、内容面板底色、选中高光、悬停高光、半透明遮罩和详情框背景各自有语义令牌，禁止用“额外灰色遮罩”修正整个窗口观感，避免破坏青色 / 渐变 / 图片皮肤。
   - 推荐阶段拆分：
     - `Dark-0`：盘点所有硬编码颜色和内联 QSS，按窗口 / 弹窗 / 自绘组件建立清单。
     - `Dark-1`：补齐 Theme Tokens 与暗色 preset，不改变 UI 结构。
     - `Dark-2`：迁移主窗口、日 / 周 / 月窗口头部、搜索框、更多菜单和基础按钮。
     - `Dark-3`：迁移详情弹窗、picker、颜色弹窗、hover 预览和共享弹窗。
     - `Dark-4`：迁移日课表、周课表、坐标看板、月格 marker 等自绘组件。
     - `Dark-5`：做亮色 / 暗色 / 渐变 / 纯色 / 高 DPI 的视觉回归，确认没有白块、漏染图标、文本不可读和选中状态丢失。
   - 暗色模式当前优先级低于课表收尾、搜索 / 筛选、导出和测试基线。除非用户明确切换到皮肤 / 主题阶段，否则后续普通功能轮不再继续扩大暗色模式点修范围。

10. 完成节假日显示。

    实现前应确认：

    - 节日范围是否包含公历节日、农历传统节日和法定节假日 / 调休标记，并明确优先级和同日多标签展示规则。

    - “显示节日”开关只控制节日 / 节假日标签显示，不影响日期、日程、选中态和添加逻辑。

11. 完成字体功能。

    补齐当前日 / 周 / 月添加界面中已有或预留的字体入口，并在右上角更多菜单中提供全局字体设置入口。字体设置需要明确作用范围，不能只修改当前控件而不持久化，也不应由 UI 直接批量改库。

    添加或编辑单条日程 / 待办时：

    - 应用到当前内容：只影响当前正在添加或编辑的这一条日程 / 待办。
    - 应用到同类型全部内容：当前添加日程时应用到全部日程；当前添加待办时应用到全部待办。
    - 应用到全部内容：同时应用到全部日程和待办。

    从更多菜单设置字体时：

    - 应用到当前页面对应类型：当前处于日程页面时应用到全部日程；当前处于待办页面时应用到全部待办。
    - 应用到全部内容：同时应用到全部日程和待办。
    - 更多菜单入口不提供“仅当前单条”范围，因为该入口没有明确的单条内容上下文。

    实现前应先审查：

    - 当前日程和待办模型是否已有字体显示字段；单条字体设置若需新增字段，必须使用显式迁移并保持旧数据默认显示兼容。
    - 同类型和全部内容的字体设置优先作为显示偏好保存，不应通过逐条批量改写历史记录模拟全局设置。
    - 明确字体可配置项、默认值、预览、确认和取消语义；取消后不得污染当前表单或全局偏好。
    - 日界面、周界面、月界面持久弹窗、待办列表和待办看板应读取同一字体策略，避免各窗口独立维护样式状态。
    - 字体策略应为后续换肤和云同步保留统一设置接口，但本阶段先完成本地持久化与跨窗口刷新，不依赖云服务。

12. 本地数据库加密与本地多账号。

    该方向属于日程隐私增强的特殊需求，优先级暂低，但应安排在云同步之前，避免云同步阶段再次重做账号、数据库路径和密钥边界。更多菜单中的“安全加密”当前仅为 UI 占位壳，不代表真实加密已经实现。

    第一阶段建议采用 SQLCipher 整库加密，复用 `sqlcipher3` 与 Peewee `playhouse.sqlcipher_ext.SqlCipherDatabase`，不自行实现数据库加密算法。现有 `db_manager` 在模块导入阶段立即连接并建表，实施前必须先改为延迟初始化：应用确认当前本地账号和密码后再绑定数据库、连接、建表和运行迁移，未启用加密时继续使用现有明文 SQLite 行为。

    明文转加密、加密转明文和更换密码必须分别设计：

    - 明文 / 密文转换使用临时目标数据库和 `sqlcipher_export()`，验证表结构、关键记录数和基础查询后再原子替换；失败时保留原数据库并完整回滚。
    - 更换加密密码使用 SQLCipher 既有 rekey 能力，不通过逐条读写日程模拟重加密。
    - 密码、数据库密钥和派生结果不得明文写入配置、日志或数据库；配置只保存加密状态、数据库格式版本、KDF 参数版本、账号 ID 和数据库路径等非秘密元数据。
    - PBKDF2 / SQLCipher KDF 的职责必须在技术验证阶段固定，避免应用层派生后又被 SQLCipher 当作口令重复派生；参数需要在目标 Windows 设备上做耗时测试并版本化。
    - Python 运行时无法承诺密钥从进程内存、分页文件或崩溃转储中绝对擦除，安全说明不得写成“密钥只存在绝对安全内存”。

    启动解锁建议使用独立 `UnlockWindow`，密码验证和数据库初始化成功后再创建 `MainWindow`。信封、V 形折口、锁 / 锚 / L 标识和 `QPropertyAnimation` 解锁动画属于后续视觉层，不应与数据库迁移和密码校验写在同一工单。挂起恢复需先明确是仅遮挡 UI 的隐私锁，还是关闭数据库连接、清空 UI 缓存并暂停提醒查询的完整锁定；错误次数、递增等待、五次退出及通知内容隐藏规则需单独确认。

    本地多账号放在整库加密之后实现，优先采用“一账号一数据库”：本地 profile 注册表只保存账号 ID、显示名称、数据库路径和格式版本，各账号分别持有独立明文或加密数据库。切换账号时必须关闭当前连接、清理窗口 / repository 缓存，再初始化目标数据库；全局主题等应用设置与账号级日程设置需明确分层。该结构应为后续云账号保留稳定 profile ID，但本阶段不实现服务端、设备同步、找回密码或云端密钥托管。

    实施前先做隔离技术样本，不直接修改真实 `schedule.db`：验证 Python 3.11 Windows x64 下 `sqlcipher3` 安装、明文导出到加密库、正确 / 错误密码打开、迁移失败回滚、重启重连和最终打包依赖。样本通过后再拆分数据库生命周期、迁移、启动解锁、挂起锁定、动画和多账号工单。

13. 云功能。

    最后单独规划账号、设备、同步、冲突解决、服务端和安全策略。

可选增强：

- 天气模块可后续单独处理自动重试、缓存上次成功天气、备用天气源或错误详情面板；基础失败兜底已完成，不再作为主线任务。
- 待办看板文字适配已完成当前小修；如后续实际 UI 仍有局部问题，再作为小修处理，不再列为主线任务。
- 更多菜单已新增“使用帮助”入口和“使用手册 / 帮助助手”二级菜单壳；真实手册内容和帮助助手能力可在云功能前后单独规划，不阻塞当前主线。

不建议：

- 在没有阶段规划的情况下直接启动坐标看板或云同步。
- 在 UI 适配阶段顺手大拆数据层、服务层或 Controller。
- 在功能阶段顺手改数据库结构。
- 在功能阶段伪实现换肤、排序、筛选、同步。
- 把视觉微调和复杂业务链路混在同一个工单中。

当前复现范围：

- 已确认在日界面和待办界面触发。
- 周界面和月界面当前未复现，不应在没有证据时扩大修改范围。

已确认触发路径：

1. 打开更多菜单后按 `Shift+Win+S`，应用失去激活状态，菜单随后可能落到主窗口后方。
2. 打开更多菜单，单击“模式切换”锁定二级菜单；随后点击其他应用的输入框使本应用失焦，再返回并再次点击“模式切换”，可触发该问题，并存在相近失焦变体。

已排除或验证不足的方案：

- 单纯给 popup 同步 `WindowStaysOnTopHint` 并在显示后调用 `raise_()` 无法解决，相关试验代码已回退。
- 该问题不是菜单初次显示位置错误，也不是父窗口移动；重点是应用失焦后 native popup 的生命周期、锁定状态和 Windows z-order 重排。

后续修复建议：

- 优先监听 `ApplicationDeactivate/WindowDeactivate`，在应用失焦时关闭主菜单及全部二级菜单，并停止 hover timer、复位 lock 和 hover 状态。
- 补齐 `mode_menu` / `help_menu` 独立隐藏时的状态清理，避免二级菜单关闭后残留锁定标记。
- 首轮保留主菜单现有 `exec()` 调用模型；若真实 Windows 复测仍失败，再单独评估改为非阻塞 `popup()`。
- 必须使用上述两条真实触发路径做 Windows 本地验收，offscreen 测试不能替代原生 z-order 验证。
